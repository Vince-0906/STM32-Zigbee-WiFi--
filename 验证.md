## 2. 烧录顺序与工程

### 2.1 CC2530 端节点（两块）
工程：`ZStack-CC2530-2.5.1a/Projects/zstack/Samples/ZbNode/CC2530DB/ZbNode.eww`

- IAR 打开，选择 Build Configuration：
  - **Node1_EndDeviceEB**：`-DNODE_ROLE=1`，DHT11 温湿度，2 s 上报
  - **Node2_EndDeviceEB**：`-DNODE_ROLE=2`，光敏 ADC，500 ms 上报
- 两块 E18-MS1 板通过 CC Debugger 分别烧录对应 hex
- 上电后 LED3（P1.0）慢闪 = 未入网，快闪 = 正在入网，熄灭且 LED4（P1.1）常亮 = 成功入网并首帧 confirm

### 2.2 CC2530 协调器
工程：`ZStack-CC2530-2.5.1a/Projects/zstack/Samples/GwCoord/CC2530DB/GwCoord.eww`

- Build Configuration：`CoordinatorEB`
- 焊接/插接在 STM32 板的 ZigBee 插座（UART0-Alt1 P0.2/P0.3 → PC10/PC11，RESET 由 PC4 控制）
- 上电后：
  - LED1 (P1.0) 常亮 = Zigbee 协调器建网完成
  - LED2 (P1.1) 每次 UART 收帧/AF 收帧闪烁
  - 板载 OLED 刷出 `ZB<state> C15 P<panid>` 等信息

### 2.3 STM32 网关
工程：`STM32/ZNJJ.uvprojx` (Keil MDK)

- 编译选项里确认 `WIFI_SSID="611-2"` / `WIFI_PWD=...` / `SERVER_IP="192.168.62.35"` / `SERVER_PORT=23333`（`App/config.h` 默认值，若现场 WiFi 不同需用 `-D` 覆盖）
- 烧录后接 CH340 USB 查看 `115200 8N1` 日志串口；同时 PA15 蜂鸣器短暂鸣叫应被新代码抑制

## 3. 上机验证步骤

### 3.1 冷启动 10 s 通电检查（对应规范书 A1）

1. 依序给 **两个节点** → **协调器** → **STM32 网关** 上电
2. 观察 STM32 调试串口（PA9/PA10，115200）应 10 s 内依次打印：
   ```
   [*][I][gw] boot, f=72000000
   [*][I][wifi] state=1        # W_ENABLE
   [*][I][wifi] at> AT+WMODE=1,1
   [*][I][wifi] at< OK
   [*][I][wifi] at> AT+WJAP=...
   [*][I][wifi] at< +EVENT:WIFI_GOT_IP
   [*][I][wifi] at> AT+SOCKET=4,192.168.62.35,23333
   [*][I][wifi] at< connect success ConID=1
   [*][I][wifi] at> AT+SOCKETTT
   [*][I][wifi] at< >
   [*][I][zb]  bringup window start src=alive-edge dur=70000 retry=5000
   [*][I][zb]  tx ALLOW_JOIN sec=60 src=alive-edge
   ```
3. STM32 板载 OLED：
   ```
   GW ZB:O WF:O P:1
   N1 -- --
   N2 --
   ALM:---
   ```
   （修复 #8 后：只要 WiFi 进入透传态就显示 `P:1`，无需 PC 端主动发数据；若 WiFi 掉线会变 `WF:X P:0`）
4. CC2530 协调器板载 OLED：
   ```
   ZB9 C15 P1234        # state=9 COORD, ch=15, 随机 PANID
   N1 -- --
   N2 --
   ALM:---
   ```

**失败排查**
- 若 STM32 OLED 全黑：检查 PA11/PA12 是否被 SWD 复用宏占用（本工程只禁了 JTAG 不动 SWD）
- 若 WiFi 卡在 `state=3 W_WJAP`：检查 SSID/密码、AP 信号；模块不支持的字符需换纯 ASCII
- 若 WiFi 已上但日志一直 `backoff`：看 `last_tx`/`last_rx`；`192.168.62.35:23333` 必须提前运行 PC 工具监听

### 3.2 PC 上位机连线（TCP 客户端模拟）

STM32 是 TCP **客户端**，PC 上起 TCP **服务器**监听 `23333`。Windows 下可用 Python：
```bash
python -c "import socket; s=socket.socket(); s.bind(('0.0.0.0',23333)); s.listen(); c,_=s.accept(); \
           import threading; threading.Thread(target=lambda: [print(c.recv(4096).decode(errors='ignore'),end='') for _ in iter(int,1)]).start(); \
           while 1: c.sendall((input()+'\n').encode())"
```
或用 NetAssist、TCP Tools 等工具。连上后 STM32 会立即发：
```json
{"t":"hello","fw":"znjj-0.1","gw_id":1}
```

### 3.3 温湿度 Node1 持续上报（A2）

1. 插电 Node1（DHT11 实物），约 5 s 内 OLED 第二行出现：
   ```
   N1 T:26.34 H:58
   ```
2. PC 端每 ~2 s 收到一行：
   ```json
   {"t":"report","node":1,"kind":"temp_hum","temp":26.34,"hum":58,"ts":...}
   ```
3. 用手握住 DHT11 使数值上升，OLED 与 PC 数据同步变化
4. **本次修复 #2 验证**：把 DHT11 放到冰箱短暂冷却至负温（如 -0.5 °C），PC 端应打印 `"temp":-0.50` 而非 `"temp":0.50`（老固件的 bug）

### 3.4 光敏 Node2 持续上报（A3）

1. 插电 Node2，OLED 第三行出现 `N2 L:<数值> lx`
2. PC 端每 ~500 ms 收到一行 `{"t":"report","node":2,"kind":"lux","lux":...}`
3. 用手遮住 GL5528，lux 值应下降到 100 以下；打开手电直射，应上升到 1500+

### 3.5 远程开/关灯（A4）

从 PC 发送：
```json
{"t":"cmd","seq":42,"node":1,"target":"led","op":"on"}
```
期望 ≤ 1 s 内：
1. Node1 板上 LED1（P1.4）亮
2. PC 收到：
   ```json
   {"t":"ack","seq":42,"ok":true}
   {"t":"status","node":1,"led":"on","buzzer":"off","ts":...}
   ```
**本次修复 #3 验证**：`"ok":true` 必须是 JSON **布尔**（不带引号）。若 PC 端用严格 JSON 解析器（`json.loads`）不会抛异常即为通过。

### 3.6 toggle 操作（本次修复 #4）

连续发两次：
```json
{"t":"cmd","seq":101,"node":1,"target":"buzzer","op":"toggle"}
{"t":"cmd","seq":102,"node":1,"target":"buzzer","op":"toggle"}
```
期望：
- 第一次 Node1 蜂鸣器鸣响，`status.buzzer="on"`；30 s 内自动联动处于 override，不会自动关闭
- 第二次蜂鸣器静默，`status.buzzer="off"`
- 老固件会回 `{"ok":false,"err":"op"}`，新固件两次都回 `{"ok":true}`

### 3.7 光照自动联动（A5 + A7）

1. 默认阈值 `lux_low=500`。用黑布遮住 Node2 使 lux < 500 并保持 > 1 s
2. 期望：
   - Node2 LED 自动亮
   - PC 收到 `{"t":"alarm","type":"light","level":"on","val":<lux>,"threshold":500,"ts":...}`
   - OLED 第四行：`ALM:light-ON`
3. 揭开黑布让 lux > `500 + hysteresis_lux(50)` = 550 保持 > 1 s：
   - Node2 LED 自动灭
   - PC 收到 `{"alarm","type":"light","level":"off",...}`
   - OLED 第四行恢复 `ALM:---`

### 3.8 温度自动报警（A6）

用电吹风对准 DHT11 加热至 `> 32 °C` 保持 > 1 s：
1. Node1 蜂鸣器响
2. PC 收到 `{"t":"alarm","type":"temp","level":"on","val":3250,"threshold":3200,...}`
3. 降温后 Node1 重回 `[5 °C, 31.5 °C]` 带滞回区间再持续 1 s，蜂鸣器关闭且 alarm off

### 3.9 阈值持久化（对应 §5.6）

PC 端发：
```json
{"t":"set_threshold","seq":200,"lux_low":100,"temp_high":40}
```
期望：
- PC 收 `{"t":"ack","seq":200,"ok":true}`
- 下次遮光需要到 lux < 100 才会开灯（立即生效）
- **断电再上电**后阈值仍然保留（Flash 末页持久化）。可从串口观察首次启动没有 `cfg defaulted` 警告

**失败排查**
- 若返回 `{"ok":false,"err":"nv"}`：Flash 末页擦写失败，一般是因为芯片实际容量与 `0x1FFFF7E0` 读出的不符。已在 `drv_flash_config_page_addr()` 做 RBT6/RCT6 自适应；如仍不行可临时 hardcode 到 `0x0803F800`（RCT6 256K 末页）

### 3.10 节点离线（A8）

1. 拔 Node1 电源 → 等待 ≥ 3 min（`ZB_HEARTBEAT_S * 3 = 180 s`）
2. 期望：
   - 协调器 OLED `N1 -- --`
   - STM32 OLED `N1 -- --`
   - PC 收到 `{"t":"node_info","node":1,"online":0,...}`
   - 若此前 Node1 蜂鸣器正在报警，自动撤回 → 收到 `alarm temp off`

### 3.11 PC 断网 30 s（A9 + 本次修复 #5）

1. 正常透传中，在 PC 端直接关闭 TCP 服务器程序
2. STM32 的 `W_TRANSPARENT` 状态下 **30 s** 内没有任何下行字节 → 日志打印：
   ```
   [*][W][wifi] transparent idle >30000ms, reconnect
   [*][W][wifi] backoff=2s
   ```
3. 2 → 4 → 8 → 16 s 指数退避后重新建连
4. **断网期间**，本地联动（光照遮光、温度越限）仍应正常工作，Node 数据缓存继续更新，OLED 仍刷新
5. 重新打开 PC 服务器后 STM32 会再次发 `hello` 并恢复上报

**失败排查**：若 30 s 后 STM32 不重连，确认：
- `config.h` 里 `TCP_DEAD_MS = 30000u` 未被修改
- `wifi_uart_read()` 有字节返回时 `s_last_rx_ms` 能被更新到 `ms_now()`

### 3.12 CC2530 看门狗/冷复位恢复（本次修复 #6）

**模拟 CC2530 掉线：**
1. 运行过程中拔掉 CC2530 模块或短接其 RESET 引脚保持 > 30 s
2. STM32 日志每 10 s 看到一次 PING 发送，但无回应
3. 30 s 后（3 次 miss）自动：
   ```
   [*][W][zb] heartbeat lost, resetting CC2530
   ```
4. PC4 拉低 ~1 ms 让 CC2530 冷启动；CC2530 上电→建网→发首帧 → `s_alive` 恢复
5. 同步触发 60 s 入网窗 → `bringup window start src=alive-edge`

**冷启动从未就绪场景（本次修复 #6 核心）：**
- 拔掉 CC2530 的情况下直接给 STM32 上电：老固件因为 `s_last_rx_ms==0` 会**永远不重置** CC2530。新固件初始化时写入当前 ms，3×10 s 后会尝试复位一次（即便模块是拔掉的，后续插回也会很快被重新拉起）

### 3.13 72 h 老化（A10）

保持以上所有节点持续运行 72 小时。期望：
- STM32 调试串口无持续的 `ERR` 或 `backoff` 循环
- PC 端数据连续，无中断 > 1 min
- `zb_link_err_cnt()` 相对 `zb_link_ok_cnt()` 比例 < 1%
- 无 IWDG 触发的冷复位（可通过 `RCC_GetFlagStatus(RCC_FLAG_IWDGRST)` 在 boot 日志中检查；本项目未显式打印但可加）

## 4. 快速故障表

| 现象 | 可能原因 | 排查方向 |
|---|---|---|
| CC2530 协调器 OLED 全黑 | 本次修复 #1 (b)/(c)/(d) 没编进去 / 或实际焊错 P1.2/P1.3 | 重新确认 `soft_i2c.c` 中 `OLED_SCL_BM=0x04 (P1.2)`、`OLED_SDA_BM=0x08 (P1.3)`；用万用表或示波器量 P1.2/P1.3：启动瞬间应看到 ~100 kHz 方波约 1 s；若完全无信号，多半 `P1SEL` 被 USART/Timer 抢走；若有信号但不亮，多半 3.3V 供电或 U10 接触不良 |
| STM32 OLED 全黑 | PA11/PA12 未初始化或被其它复用占掉 | `drv_oled_init()` 是否被 `oled_view_init()` 调用；SWD 仍在 `PA13/14`，不会冲突 |
| WiFi 连不上 | SSID/密码/AP 兼容性 | 把 `WIFI_SSID`/`WIFI_PWD` 改成手机热点测 2.4G；日志里 `at< FAIL` 往往是密码错 |
| PC 看不到数据 | 防火墙 / IP 不通 | `telnet 192.168.62.35 23333` 先通；STM32 日志出现 `connect success ConID=1` 表示连上 |
| Node 数据不更新 | 未入网 / 对应 NODE_ROLE 没烧对 | Node LED3/4 状态；协调器 OLED 顶行 `ZB` 状态数字应为 `9 DEV_ZB_COORD`，节点侧 `DEV_ROUTER/DEV_END_DEVICE`（6/7） |
| 负温 PC 显示成正数 | 本次修复 #2 未编进去 | 在 `json.c` `json_add_fp2` 应能看到 `sign = "-"` 逻辑 |
| `toggle` 被拒 | 本次修复 #4 未编进去 | `router.c` `on_cmd_json` 应 `strcmp(op,"toggle")==0` 分支 |
| 30 s 无下行不重连 | 本次修复 #5 未编进去 | `wifi_link.c` 应有 `transparent idle >` 日志 |
| OLED 始终 `P:0` 即便 PC 端已收到 JSON | 本次修复 #8 未编进去（老固件 `pc_ok` 依赖 `s_last_pc_rx_ms != 0`） | 看 `gw_main.c` 是否改用 `wifi_up ? 1u : 0u` 作为 P 标志；PC 端若不回任何包也能 `P:1` |
| 外接 DC 上电蜂鸣器常鸣 | 本次修复 #9 未编进去（PA15 JTDI 默认上拉高、板上 Q1 实为高有效） | 检查 `drv_gpio.c` 里 `GPIO_ResetBits(GPIOA, GPIO_Pin_15)` 是否在 `GPIO_Init` 前后各一次，`buzzer_set(1)` 是否走 `SetBits`；实测 USB 供电下也应静默，ALM 激活时才鸣叫 |
